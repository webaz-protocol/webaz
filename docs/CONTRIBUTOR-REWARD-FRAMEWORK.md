## §0 本文档的性质声明 / What this is

**本方案是**:一份目标声明 + 演化框架。锁定"我们要达成什么"(目标层),以及"由谁、以什么流程,在未来决定怎么做"(演化层)。

**本方案不是**:终态机制。它不写死分配比例 / 曲线 / 结构;不假装创始人此刻一个人能拍板完整答案;不假装预见了所有未来的法律 / 市场 / 技术变化。

**为什么这样设计**:创始人此刻的认知,必然不如未来的专业团队(法律从业者、经济学家、各辖区合规专家、可能的政府指导)。把具体机制写死 = 用今天的有限认知锁死未来的可能性。成熟的做法是钉死目标,把机制的决定权完整交给未来。

```
This framework locks the GOAL and the PRINCIPLE.
It does NOT lock any concrete distribution mechanism.
All parameters / curves / forms are decided by the phase-D DAO +
professional team (incl. legal practitioners, gov guidance if needed),
and evolve continuously.
```

## §1 目标层(锁定 / 不可演化)/ Goal Layer (locked)

```
目标 1:贡献多的人,占据重要席位
目标 2:贡献,得到应有的回报
目标 3:治理与经济,统一、优雅、可持续

这四条是 WebAZ 的立项初心,锁定。怎么衡量、怎么兑现是机制层(可演化);"贡献多者得更多"这个方向是目标层(锁定)。

## §2 核心原则:回报锚定"真实贡献的累积量 × 衰减"/ Core Principle

这是本框架的灵魂,也是它区别于"资金盘 / 微商盘"的根本。

```
回报 = f(真实贡献的累积量 × 生命周期衰减)
Reward = f(cumulative real contribution × lifecycle decay)
```

### 2.1 三条不可动摇的子原则

```
① 回报锚定"真实贡献",不锚定"人头 / 位置 / 来得早"
   - 真实贡献 = 真实交易、代码、履职(仲裁/验证)、传播带来的真实成交
   - 不靠"发展下线人头"获利
   - 推荐了人,光凭人头本身,得不到利益

② 早行动者凭"累积量大"占优,不凭"早"本身得利
   - 早行动的人,因为有更长的时间窗口去累积真实贡献 → 累积基数大 → 回报大
   - 但回报的【因】是"贡献",不是"早"
   - "早"只是常与"大贡献"相关,不是回报的来源
   - 一个很早注册却从不使用、从不分享、零贡献的人 → 累积量=0 → 回报=0,且【不该得利】

③ 历史贡献随时间 / 被替代而衰减,防"躺在功劳上吃一辈子"
   - 三年前带来的成交,对"今天的平台"贡献在衰减
   - 今天带来的成交,对"今天的平台"是满额
   - 严重劣迹衰减慢,轻微劣迹衰减快(同一衰减框架的镜像)
   - 结果:早行动者总账仍占优(累积基数大),但不能靠陈年旧功永久抽成

④ "立项"和"持续维护"是贡献类型,不是身份特权
   - 把协议从零立起来(立项)+ 持续审核合并/定方向/运维(维护)= 真实贡献(详 §4.2)
   - 它们按"做了什么"计入,不按"是谁"
   - 创始人因同时做了立项+维护+代码,真实贡献累积量本来就该很高 —— 这是【算出来的】
   - 明确拒绝:任何"凭创始人身份的收益保底/底线"(那是身份特权,违 #5 不偏袒)
   - 检验:若晚来者做出同等贡献,同一把尺子给他同等回报 → 对所有人一致
```

### 2.2 一句话

```
早起的鸟儿有虫吃 —— 但前提是它真的去抓了虫。
The early bird gets the worm — but only if it actually catches one.
```

### 2.3 检验尺子(供未来决策者参考,非强制)

```
"未来加入的强贡献者,能否凭真实贡献,
 获得与其贡献相称的回报和地位?"

能 = 健康的精英制(早行动者凭累积占优,但晚来强者可凭贡献超越)
不能 = 有毒的世袭制(老人凭位置/资历永久把持)
```

## §3 关系层 / 估值层分离(结构性原则)/ Relationship vs Valuation Layer

本框架唯一的结构性约定,它保证未来有最大的演化自由。

```
关系层(Relationship Layer)— 从第一天起如实记录,不可逆:
  - 每个 webazer 在关系网络(二叉树)里的位置 = 系统记录的坐标
  - 真实推荐关系(谁通过谁的链接来,若有)
  - 这层是"历史事实",像账本:只追加,忠实记录
  - 不附带任何"价值判断 / 收益权重"
  - 全员如实记录(包括未参与奖励的纯消费者),保证后期 DAO 数据完整

估值层(Valuation Layer)— 后期 DAO + 专业团队定义,持续演化:
  - 如何把"关系层的事实 + 贡献层的信号"换算成"回报"
  - 维度、权重、曲线、形式 —— 全部在这一层,全部可演化
  - phase A 不定义具体公式;留给未来

为什么这样分:
  - 关系层先记录(不记录 = 未来无数据可用)
  - 估值层后定义(此刻定义 = 锁死未来)
  - 数据从第一天攒,机制由未来最懂的人定
```

### 3.1 位置 ≠ 收益权 / Position is NOT entitlement

**位置不能是独立收益源**。"占位就有收益"违反 §2.1 子原则①(回报锚定真实贡献,不锚定位置 / 来得早 / 人头)。

**Position cannot be an independent income source**. "Hold a position → get income" violates §2.1 sub-principle ① (reward anchored to real contribution, not to position / arrival time / headcount).

- ❌ 早占位 = 自动每月发钱 / Holding early position = automatic monthly payout
- ❌ 树根附近 = 凭位置抽下层每一单 / Near-root = take a cut of every downstream transaction by position alone
- ❌ 位置预留 = 无贡献也按位置预分 reward / Pre-allocated position slots = pre-reward without contribution

### 3.2 位置可以是收益公式的【修饰参数】 / Position CAN be a modifier in reward formula

**位置可以作为收益公式的一个加权 / context 参数,但有 4 个边界约束**:

**Position MAY be a weighting / context parameter in the reward formula, with 4 boundary constraints**:

```
数学模型 / Mathematical model:

reward = f(contribution) × g(position) × h(decay) × ...
         ↑ base               ↑ modifier      ↑ time
         必须 > 0             由 DAO 定权重    framework §2.1③
         (零贡献 → 零回报)
```

**4 条边界约束 / 4 boundaries**:

1. **乘法不是加法** / Multiplication, not addition
   - `reward = contribution × position_weight`(可) / OK
   - `reward = contribution + position_bonus`(不可,位置 bonus 独立于贡献) / Forbidden

2. **base 必须 > 0**:零贡献 × 任何位置 = 零收益 / Zero contribution × any position = zero
   - 这是 §3.1 的数学化:位置不能凭空生钱

3. **position_weight 区间由 DAO 定**,且不能压过 base
   - phase A 默认 `g(position) = 1`(完全不用,等于纯贡献回报)
   - DAO 可设区间 [0.5, 2.0] 等;不能允许"高位置 × 极小贡献 ≥ 低位置 × 中等贡献"反转
   - 任何 position_weight 公式必须公开可验证(对应 #2 + #9)

4. **非可继承**:位置加权不能传给"占位但不贡献"的后代节点
   - 这关键防止"位置预留" / 位置占坑套利
   - 详 §3.3 关系网络遗产边界

### 3.3 关系网络遗产边界 / Network inheritance boundary

关系层记录了"谁带来谁",但**回报不在关系层世袭**:

- 你的上线 / 你介绍来的人,不会因为存在而自动给你发钱
- 你的位置可以是修饰参数(§3.2),但 base 仍必须是你自己的真实贡献
- 若你长期不贡献,你的位置 modifier 不会让你"凭老资格吃下线"
- 反例:若一个 retired 用户位置 = root + position_weight=2 → 多年不贡献仍持续拿钱 = 违反 §2.1③ 衰减原则

**关系网络是数据,估值是机制,衰减是时间**。三者各自独立,DAO 演化各自参数。

### 3.4 类比 / Analogies

| 系统 | base(必须) | position modifier(可选) | 零 base 结果 |
|---|---|---|---|
| 学术 H-index | 引用数 | 跨场域 / 时效权重 | 无引用 = 无 H 分 |
| GitHub stars | star 数 | 仓库时间 / 主题热度 | 无 star = 无影响力 |
| 二叉树位置 + 贡献(本框架) | 真实贡献 | 位置 / 衰减 / DAO 设权重 | 零贡献 = 零回报,无视位置 |

## §4 贡献层:六类真实贡献(现在建,持续累积)/ Contribution Signals

回报锚定的"真实贡献",来自这六类(度量维度参数交未来 DAO):

```
1. 交易贡献      真实买卖(GMV / 完成订单)
2. 代码贡献      新写代码 / 集成 / agent —— 见 §4.1 度量探讨
3. 履职贡献      仲裁 / 验证 / 治理参与(质量,见 GOVERNANCE-LEADERBOARD-SPEC)
4. 传播贡献      内容 / 推广,真实带来的成交(per-order 显式归因)
5. 立项贡献      "无中生有把协议立起来" —— 见 §4.2(一次性,但权重不低)
6. 维护性贡献    审核合并 / 定方向 / 答疑 / 运维 —— 见 §4.2(持续,常被忽视)
```

> 第 5、6 类是本框架特意补上的——传统度量常只算"写新代码 / 做成交",忽视了"把项目从零立起来"和"持续维护让它不散架"这两类真实且关键的贡献。它们不是身份特权,是被低估的贡献类型(详 §4.2)。

### 4.1 代码贡献度量(探讨,具体公式交 DAO)

代码"从出现到被替代"是大概率事件,所以代码贡献该用"生命周期资产"视角度量,不是"一次性记功":

```
价值侧(难伪造的金指标):
  使用频次 × 存活时长 × 当前占比
  - 使用频次:真实生产环境调用(难伪造,最接近真实价值)
  - 存活时长 / 生命周期:借鉴 git-of-theseus 衰减曲线
  - 当前占比:按"当前仍在运行的代码"算,不按"历史提交"算
    → 被重写替代的代码不再计占比(接 §2.1③ 衰减)

成本侧(做分母,不做分子):
  token 投入 —— 用作"效率分母",不是"回报分子"
  → 高效率 = 低成本造高价值 → 奖励精炼,惩罚"烧 token 刷量"

质量门槛(一票否决):
  出过安全事故 / 被推翻 → 该贡献清零(连带责任)

显式【不用】"工作量 / 代码行数"作正向指标(最易刷,违背质量原则)。

前任/后任分配(借鉴学术引用网络 PageRank):
  代码 A 被代码 B 替代后,A 的历史贡献进入衰减期(非立即归零),
  B 若是 A 的演进则继承部分"地基价值";全新独立功能不给前任分配。
  半衰期 / 地基继承比例 / 演进判定 —— 参数交 DAO。
```

> 提醒(Goodhart 定律):任何公开能换回报的指标,都会被优化而非反映真实。解药是价值侧用难伪造信号 + 质量一票否决 + 具体换算交 DAO 持续对抗。也提醒:**代码贡献只是一个赛道,监管 / 安全 / 治理 / 社区贡献同样是真实贡献**,别让度量窄化成"只有码农有价值"。

### 4.2 立项贡献 + 维护性贡献(常被忽视的两类)

传统度量重"写新代码 / 做成交",轻"立项 / 维护"。但这两类是真实贡献,本框架明确计入。**它们锚定"做了什么",不锚定"是谁"——同一把尺子对所有人,不是创始人特权。**

```
立项贡献(类型 5)——"无中生有"的一次性贡献:
  - 把协议从零设计、立起来、定下初始方向 = 一项真实贡献
  - 它一次性发生(项目只被立一次),但权重不低
    (从 0 到 1 的难度,常高于从 1 到 N)
  - 度量:不是"写了多少行",是"立起来的东西后续被多少真实使用/贡献所依赖"
    → 接 §4.1 价值侧逻辑:立项成果的"被依赖度 × 存活度"
  - 衰减:同样适用 §2.1③ —— 立项的历史功劳随项目演进而部分衰减,
    不能靠"我当年立的项"永久满额抽成;但因"被依赖度"通常很高,
    总账长期占优(这是"算出来的占优",不是"保出来的特权")

  ⚠️ 关键区分(回应"创始人保底"的诱惑):
    ✅ "立项是一项贡献,按其真实被依赖度计入" —— 本框架采纳
    ❌ "创始人收益设个保底,不能低于某值" —— 本框架拒绝(那是身份特权,违 #5)
    → 创始人的高回报应该是"立项+维护+代码贡献算出来很高",
      不是"凭创始人身份保出来"
    → 若一个晚来者做了同等贡献,尺子给他同等回报(对所有人一致)

维护性贡献(类型 6)——持续的、不产生新代码行的关键工作:
  - 审核合并他人 PR / 定方向 / 答疑 / 运维 / 应急响应 / 仲裁兜底
  - 这些不产生"新代码行",但项目没它们会乱、会漂、会散
  - 传统"代码行/commit数"度量完全忽视这类 → 本框架明确计入
  - 度量(难伪造信号优先):
    · review 合并的 PR 数 × 这些 PR 后续的存活/被依赖(质量,不只数量)
    · 定方向决策被后续真实采纳 / 验证正确的程度
    · 应急响应 / 安全处理的真实次数与效果
    → 同样避免"工作量"陷阱:不是"审了多少 PR",
      是"审过的 PR 后续质量好不好"(防"乱盖章刷数量")
  - 衰减:维护是持续的,所以它持续累积,不像立项是一次性
    → 一个持续维护的人,累积量持续增长(这正是"持续参与=持续贡献")

为什么这两类对创始人尤其相关(但不限于创始人):
  - 创始人通常同时做了类型 5(立项)+ 类型 6(持续维护)+ 类型 2(代码)
  - 所以创始人的真实贡献累积量,在诚实的尺子下本来就该很高
  - 但这是"因为做了这些事",不是"因为是创始人"
  - 任何人持续做类型 6,都同样累积;任何人若未来做出类型 5 级别的
    重大立项(如开辟一个全新关键模块),也同样计入
```

> 这两类补进来,既诚实回应了"立项和持续维护是真实贡献该被回报"(对),又守住了"按贡献不按身份"的底线(不开保底特权)。创始人因此获得的高回报,是经得起共创者检验的——因为它是算出来的,且尺子对所有人一致。

## §5 phase A 当前只做的事 / What phase A actually does

```
现在(已发布,solo founder)只做:
  □ 关系层数据如实记录(二叉树坐标 + 真实推荐,代码已实现)
  □ 贡献层信号持续累积(交易/代码/履职/传播)
  □ 履职数据公示(GOVERNANCE-LEADERBOARD,observation-only,不分配)
  □ 奖励参与 opt-in(默认不参与,主动申请+知情同意,见 RFC-002)
  □ 把"未来谁来定、怎么定"的流程写清楚(本文档 §6)

明确不做:
  ✗ 不在 phase A 定分配比例 / 曲线 / 结构
  ✗ 不预设任何具体回报机制
  ✗ 不假装现在的草案是最终答案
  ✗ 上层不凭位置 / 人头自动抽下层
```

## §6 演化层:谁来定、什么时候定 / Who decides, and when

```
phase A(当前,solo):只搭框架,不做分配
phase B-C(多 maintainer + 早期专业 contributor):
  招募法律 / 经济 / 合规专业人士进入治理,起草具体机制
phase D(DAO 成立):由 DAO + 专业委员会 + 必要时政府指导,
  共同制定并持续演化完整回报机制

决定流程:
  - 走 CHARTER §6 + RFC 公示 + 多签
  - 重大经济机制引入 = meta-rule track(60d 公示)
  - 可持续修订,不是一次定终身

⚠️ phase A → D 的过渡条件(待 RFC 写明,本框架留钩子):
  "交给 DAO" 不能是无限期空头支票。必须有明确的、可验证的
  过渡触发条件(如:真实贡献者数量 / 治理成熟度指标达标 →
  DAO 正式接管 reward distribution 决策权)。
  否则 "DAO 定" 会退化为 "创始人永远说了算"。
  → 此过渡条件由后续治理 RFC 定稿。
```

## §7 ECONOMIC-MODEL §11 对齐声明 / Alignment with §11

```
本框架 SUPERSEDES ECONOMIC-MODEL.md §11 中关于"具体分润机制"的
一切表述,特别是任何"二叉树位置越靠近 root → 分润越高"的措辞。

理由:
  - "位置 = 分润" 违反本框架 §2(回报锚定贡献,不锚定位置)

  - 它与 GOVERNANCE-LEADERBOARD-SPEC §0(observation-only,非分配)冲突
  - 它与 RFC-002(默认不参与奖励)冲突

→ ECONOMIC-MODEL §11 应改为:引用本框架,声明"具体机制交未来 DAO"
